# 智格生境 / TutorGrid PPT 展示内容策划

> 用途：这份文档给 PPT 生成 agent 或负责制作 PPT 的同学使用。  
> 目标：即使 PPT agent 看不到代码，也能根据本文档生成一套完整、详细、逻辑连贯的项目展示 PPT。  
> 建议页数：24-30 页。  
> 建议展示时长：10-15 分钟。  
> 展示对象：比赛评委、老师、项目答辩观众、不了解代码实现的产品/技术评审。

---

## 一、整套 PPT 的叙事主线

### 核心一句话

智格生境是一张可以和 AI 共写、共学、共执行任务的智能书桌。

### 推荐开场叙事

传统学习中，学生往往需要在多个工具之间来回切换：资料在文件夹里，笔记在文档里，AI 对话在聊天窗口里，代码实验又在 IDE 或命令行里。  
这些工具各自强大，但学习过程本身被割裂了。

智格生境要解决的不是“再做一个 ChatBot”，而是把文档、对话、知识库、任务执行和学习记忆统一到一个学习工作台中。学生仍然是主驾，AI 是副驾。学生负责理解、判断和选择方向，AI 负责检索、整理、执行和沉淀。

### 展示时要反复强调的三句话

1. 我们不是做一个单纯聊天机器人，而是做一个 AI 学习工作台。
2. AI 不只是回答问题，还能基于课程资料引用、执行任务、生成结果。
3. 学习过程会沉淀为文档、对话、任务、产物和长期记忆。

---

## 二、PPT 总体结构建议

建议分为 6 个章节：

1. 项目背景与产品定位
2. 核心用户体验
3. 系统架构与技术路线
4. 关键能力展示
5. 当前完成度与验证
6. 总结与未来规划

推荐页数分配：

| 章节 | 页数 | 作用 |
|---|---:|---|
| 项目背景与产品定位 | 4 页 | 让评委理解为什么要做 |
| 核心用户体验 | 5 页 | 让评委看到产品长什么样、怎么用 |
| 系统架构与技术路线 | 7 页 | 展示技术含量 |
| 关键能力展示 | 7 页 | 展示能跑、能查、能执行 |
| 当前完成度与验证 | 4 页 | 展示可信度和工程进度 |
| 总结与未来规划 | 3 页 | 收束价值与后续方向 |

## 第 2 页：学习过程中的真实痛点

### 页标题

学习不是缺少工具，而是工具之间彼此断裂

### 页面可用文字

学生在学习和完成课程任务时，常见的工作流是割裂的：

- 课程资料在文件夹里
- 学习笔记在文档里
- AI 问答在聊天窗口里
- 代码实验在 IDE 或命令行里
- 结果图片和中间文件又散落在不同目录

这些工具都能解决一部分问题，但很难形成一个连续的学习过程。

### 页面重点短句

资料、笔记、对话、执行、结果无法自然沉淀到同一个学习上下文中。

### 画面建议

做一个“割裂工具链”示意图：

```text

     \             |             |          |           /
      \            |             |          |          /
       ------------ 学习上下文断裂 ------------
```

### 讲稿建议

我们观察到，学生不是没有工具，而是工具太分散。资料、笔记、AI 对话、代码实验和实验结果往往存在不同地方。AI 回答可能很有用，但它很难自然回到学生的学习文档中；代码任务可以跑出来，但过程和结果又很难沉淀为学习记录。

## 第 3 页：我们要解决的问题

### 页标题

把学习过程重新组织起来

### 页面可用文字

智格生境希望解决三个核心问题：

1. 学习资料如何进入 AI 上下文？
2. AI 回答如何沉淀到文档和学习记录？
3. 复杂任务如何从一句自然语言指令变成可执行过程？

### 页面重点短句

从“问 AI 一个问题”，升级为“和 AI 一起完成一个学习任务”。

### 画面建议

使用三段式对比：

```text
过去：问 AI → 得到回答 → 手动复制整理 → 手动跑实验

现在：写文档 → AI 引用资料 → 编排任务执行 → 结果回写文档
```

### 讲稿建议

我们的目标不是让 AI 替代学生学习，而是让 AI 成为副驾驶。学生提出目标、判断结果、选择方向；AI 负责检索资料、整理内容、执行重复任务、保存过程和结果。

## 第 4 页：产品一句话定位

### 页标题

智格生境是什么？

### 页面可用文字

智格生境是一张可以和 AI 共写、共学、共执行任务的智能书桌。

它由三个核心部分组成：

- Hyperdoc：学生和 AI 共同编辑的学习文档
- Chat：围绕当前文档和课程资料的多会话助理
- Orchestrator：能执行多步任务的 AI 编排引擎

### 页面重点短句

文档是学习主战场，Chat 是副驾驶，编排引擎负责把任务真正做完。

### 画面建议

画一个三角关系：

```text
              Chat 助理
                 / \
                /   \
               /     \
        Hyperdoc ---- Orchestrator
       学习文档        编排执行
```

### 讲稿建议

整个产品可以理解为三部分：Hyperdoc 是学生学习和沉淀内容的地方；Chat 是围绕当前文档和课程资料展开的对话助理；Orchestrator 是背后的任务执行引擎，可以把一句“帮我跑实验”的自然语言指令变成真实的执行过程。

## 第 5 页：产品信息架构

### 页标题

从工作区到文档，再到任务和对话

### 页面可用文字

产品采用三层信息架构：

```text
Workspace 工作区
  └─ Hyperdoc 学习文档
       ├─ 文档内容
       ├─ ChatSessions 多会话对话
       ├─ TaskCards 编排任务卡
       └─ Artifacts 结果产物
```

Workspace 管理课程或项目级资料。  
Hyperdoc 承载具体学习文档。  
ChatSession 保存围绕当前文档的独立对话。  
TaskCard 展示 AI 编排任务的执行状态和结果。

### 画面建议

做成树形结构或产品导航图。左边是 Sidebar，右边是 Hyperdoc 页面。

### 讲稿建议

我们用 Workspace 组织一个课程或一个项目，用 Hyperdoc 承载具体学习内容。每个 Hyperdoc 可以有多个 Chat 会话，也可以注册多个任务。这样学生不是在一个孤立聊天框里学习，而是在一个稳定的工作空间里持续学习。

## 第 7 页：Hyperdoc 文档共写

### 页标题

Hyperdoc：学生和 AI 共同编辑的学习文档

### 页面可用文字

Hyperdoc 不是普通笔记。它允许 AI 的回答成为文档结构的一部分。

核心能力：

- 学生可以直接书写学习笔记
- 选中文字后让 AI 解释、总结、改写
- AI 回答以文档气泡或引用块形式插入
- Chat 中有价值的回答可以主动插入文档
- 文档中的 AI 内容可以继续发送到 Chat 中讨论

### 页面重点短句

AI 不只是旁边的聊天窗口，而是文档共同作者。

### 画面建议

展示文档中有普通段落、AI 气泡、引用块的效果。

### 讲稿建议

传统 ChatBot 的回答很容易消失在聊天历史里。我们的 Hyperdoc 允许 AI 内容直接成为文档的一部分。学生可以对选中的文字发起 AI 操作，也可以把 Chat 中的回答插入文档，让学习过程真正沉淀下来。

## 第 10 页：典型使用流程

### 页标题

一次完整学习任务如何发生？

### 页面可用文字

一个典型流程如下：

1. 学生创建课程工作区
2. 导入课程 PPT、PDF、DOCX 等资料
3. 系统解析资料并建立知识库

6. 学生用自然语言注册编排任务
7. Worker 执行代码或外部任务
8. 系统把结果、图片、stdout 和任务状态回写到工作台

### 画面建议

做成横向流程图。

### 讲稿建议

这一页展示的是产品闭环。学习不是从一个问题开始、到一个回答结束，而是从资料进入系统开始，到文档、问答、执行、结果沉淀形成闭环。

## 第 11 页：整体技术架构

### 页标题

三通道后端架构

### 页面可用文字

系统前后端通信被拆成三条通道：

```text
Electron + Vue + TipTap 前端
        │
        ├─ SSE：/api/chat/stream
        │   普通问答、RAG、联网搜索、流式输出
        │
        ├─ WebSocket：/ws/orchestrator
        │   多步任务编排、Worker 委派、中断恢复
        │
        └─ REST：/api/*
            工作区、知识库、画像、配置、持久化 CRUD
```

### 页面重点短句

不同类型的交互使用不同通信通道，避免把所有复杂度塞进一个接口。

### 画面建议

架构图：左侧前端，右侧三个后端入口，下方连接数据库、知识库、worker。

### 讲稿建议

我们没有把所有能力都放进一个聊天接口。简单问答走 SSE，复杂任务走 WebSocket，稳定数据走 REST。这样每条链路的职责更清晰，也更容易测试和维护。

## 第 12 页：前端技术栈

### 页标题

桌面优先的前端架构

### 页面可用文字

前端采用：

用技术栈 Logo 矩阵，或者分层图：

```text
Electron
  └─ Vue 3 + TypeScript
       ├─ TipTap 文档
       ├─ Vuetify UI
       └─ Pinia 状态
```

### 讲稿建议

我们选择桌面端优先，是因为学习资料和生成产物很多都在本地。Electron 可以让前端选择本地工作区，也可以自动拉起 Python 后端和编排后端。

## 第 13 页：后端模块划分

### 页标题

后端不是单一服务，而是一组可组合能力

### 页面可用文字

后端核心模块包括：

- `server`：HTTP / SSE / WebSocket 协议入口
- `runtime`：LangGraph 编排运行时
- `sessions`：会话状态与 snapshot
- `runners`：外部请求到执行后端的路由
- `workers`：Codex / OpenCode / Python Runner 委派执行
- `knowledge`：课程知识库与文件摄取
- `rag`：课程知识问答
- `memory`：会话压缩与长期记忆
- `storage` / `db`：SQLite 持久化

### 页面重点短句

协议、运行时、知识库、记忆和执行后端分层清晰。

### 画面建议

做模块分层架构图。

### 讲稿建议

后端按职责拆成多个模块。Server 负责协议，Runtime 负责任务流转，Knowledge 和 RAG 负责课程资料，Workers 负责真实执行，Storage 负责持久化。这让系统后续扩展更容易。

## 第 14 页：Chat SSE 链路

### 页标题

Chat SSE：流式问答与工具调用

### 页面可用文字

Chat 使用 `POST /api/chat/stream`。

典型 SSE 事件：

```text
start
tool_call
tool_result
delta
done
error
```

支持三种路径：

1. 普通问题：直接调用 LLM 流式回答
2. 课程问题：优先调用 RAG
3. 最新/联网问题：优先调用 Tavily 搜索

### 页面重点短句

学生看到的是连续回答，系统内部可以自动选择 RAG 或联网搜索。

### 画面建议

左侧是请求，右侧是 SSE 事件流。

### 讲稿建议

Chat SSE 负责轻量问答场景。前端发起一次请求后，后端会持续推送事件。除了文本 delta，还会推送工具调用和工具结果，所以前端能展示“AI 正在检索课程资料”或“AI 正在联网搜索”。

## 第 15 页：Agent 路由策略

### 页标题

Agent 如何决定用不用工具？

### 页面可用文字

系统会根据问题类型做确定性预路由：

- 带课程上下文且非时效问题：优先 RAG
- 包含“最新、最近、新闻、趋势、联网、搜索”等关键词：优先 Tavily
- 没有明显工具需求：直接 LLM 回答

同时，模型也可以通过工具定义主动请求工具。

### 页面示例文字

示例 1：  
“解释一下课程里的注意力机制” → RAG

示例 2：  
“最新的正则化方法有哪些？” → Tavily

示例 3：  
“用一句话解释监督学习” → 直接 LLM

### 画面建议

做决策树。

### 讲稿建议

这里我们没有完全依赖模型自由发挥，而是加了确定性预路由。这样能保证课程问题优先使用课程资料，时效性问题优先联网搜索，提升结果稳定性。

## 第 16 页：知识库摄取链路

### 页标题

课程资料如何进入 AI 上下文？

### 页面可用文字

知识库摄取流程：

```text
上传课程文件
  ↓
解析文本 / OCR
  ↓
切分为 chunks
  ↓
生成 embedding
  ↓
写入 SQLite
  ↓
建立课程级向量索引
```

支持文件类型：

- PDF
- DOCX / DOC
- PPTX
- 图片 OCR
- 纯文本 / Markdown

### 页面重点短句

课程资料不是附件，而是可以被检索、引用和追踪的知识来源。

### 画面建议

放文件图标进入知识库，再输出 chunks 和向量索引。

### 讲稿建议

学生上传的课程文件会被解析、分块、向量化，并建立课程级索引。这样 Chat 在回答课程问题时，可以基于真实资料，而不是凭空生成。

## 第 17 页：RAG 检索增强

### 页标题

RAG：让回答有依据

### 页面可用文字

RAG 查询链路包括：

- Dense 向量召回
- Lexical 关键词召回
- RRF 融合排序
- 可选 Multi-Query

# 八、答辩时可能被问到的问题与回答

## Q1：你们和普通 ChatGPT 套壳有什么区别？

回答：

普通 ChatBot 主要解决单轮或多轮对话问题，而我们的系统把 AI 放进完整学习工作流里。它有 Hyperdoc 文档共写、课程知识库 RAG、右侧磁贴区、任务编排引擎、Worker 执行、会话持久化和长期记忆。AI 不只是回答问题，还能引用资料、执行任务、生成产物，并把结果沉淀到文档和工作区中。

## Q2：为什么需要 WebSocket 编排，直接 Chat 不行吗？

回答：

Chat 适合轻量问答，但复杂任务需要多步执行、状态推送、中断恢复和结果回传。例如“帮我跑一个线性回归 demo”不是一段回答，而是规划、运行代码、生成图片、捕获 stdout、回传 artifact 的过程。因此我们用 WebSocket 承载 `orchestrator.task.*` 任务级协议。

## Q3：RAG 的作用是什么？

回答：

RAG 让 AI 回答基于课程资料，而不是只依赖模型记忆。系统会解析课程文件，建立向量索引，回答时返回 citation。这样学生可以知道答案来自哪里，也能回到原始材料验证。

## Q4：如果本地没有 Codex 或 OpenCode 怎么办？

回答：

系统设计了 fallback 策略。Worker 不可用时不会让后端直接崩溃，而是尝试备用 worker，最终返回明确的 fallback 提示，并记录 attempted workers。Python Runner 也提供了一条稳定的本地演示链路。

## Q5：为什么选择桌面应用？

回答：

学习资料、代码和生成产物很多都在本地。桌面端可以自然访问本地工作区，也可以自动拉起 HTTP 后端和 WebSocket 编排后端，更适合“文档 + 文件 + 本地执行”的完整学习场景。

## Q6：当前还没完成什么？

回答：

主链路已经跑通，包括 Chat SSE、RAG、Tavily、编排协议、Worker 委派、Chat 持久化和工作区管理。当前主要待补齐的是 CitationTile / FileTile 的前端闭环，以及 `/task` 文档内注册入口和结果回写体验的进一步完善。
