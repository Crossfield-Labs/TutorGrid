## 一、F04 / F05 是干啥的

| 任务 | 一句话 |
|---|---|
| **F04 编排引擎 V5 适配** | 把现有 LangGraph 编排图（`planning → tools → verify → finalize`）适配到"**文档内注册任务**"模式，对前端暴露 `orchestrator.task.*` 任务级协议 |
| **F05 Worker / CLI 委派链路** | 编排引擎能把任务**委派**给可用的执行后端（Python Runner / Codex / OpenCode），失败时自动 **fallback** 不 crash |

**用户感知的效果**：
- 在 Hyperdoc 文档里 `/task 帮我跑线性回归` → 右侧编排卡 4 步进度实时更新 → 30 秒后结果回写文档 → 不会因 CLI 不可用而崩溃。

## 三、系统架构（验收前必须心里有数）

```
┌─────────────── Electron ───────────────┐
│ 前端 Vue3 (Vuetify3 + TipTap)            │
│   ├─ /landing       (Web 入口)          │
│   ├─ /board         (Electron 主板)      │
│   ├─ /hyperdoc/:id  (文档内 Chat + 编排) │
│   └─ /tasks/:id     (任务详情页)         │
└────┬───────────────┬───────────────────┘
     │               │
HTTP │              WS │ ← 编排走这条
     ▼               ▼
┌───────────────┐  ┌────────────────────┐
│ port 8000     │  │ port 3210          │
│ FastAPI       │  │ orchestrator WS    │
│ /api/health   │  │ /ws/orchestrator   │
│ /api/chat/... │  │                    │
│ /api/knowledge│  │ orchestrator.task.*│
│ /api/profile  │  │  ├ create / resume │
│ /api/config   │  │  ├ interrupt       │
└───────────────┘  │  ├ step (event)    │
                    │  ├ result (event)  │
                    │  └ awaiting_user   │
                    └─────────┬──────────┘
                              │
                    ┌─────────▼──────────┐
                    │  LangGraph runtime │
                    │  planning ↓        │
                    │  tools ↓ (delegate)│
                    │  verify ↓          │
                    │  finalize          │
                    └─────────┬──────────┘
                              │
                    ┌─────────▼──────────────────┐
                    │ Workers (F05)              │
                    │  ├─ python_runner (本地)    │
                    │  ├─ codex (CLI 分析)        │
                    │  └─ opencode (CLI 实现)     │
                    │  失败 → fallback 提示，不 crash│
                    └────────────────────────────┘
```

**两个端口都要起来**：
- `8000` — HTTP（启动后端时自动起）
- `3210` — orchestrator WebSocket（Electron 启动时自动 spawn 一个独立 Python 进程）

## 四、把环境跑起来

### 4.1 装依赖

```powershell
# 后端
pip install -r requirements.txt

# 前端
cd TutorGridFront
yarn install
```

### 4.2 启动方式 A：Electron（推荐，全自动）

```powershell
cd TutorGridFront
yarn dev
# Electron 主进程会自动 spawn：
#   1. uvicorn backend.server.http_app:app --port 8000
#   2. python -m backend.server.app --port 3210
# 启动失败会在 Electron DevTools 控制台看到日志
```

### 4.3 启动方式 B：手动两窗口（适合定位 backend 问题）

终端 1：HTTP 后端
```powershell
uvicorn backend.server.http_app:app --port 8000 --reload
# 验证：浏览器访问 http://127.0.0.1:8000/api/health 应返回 {"status":"ok"}
```

终端 2：Orchestrator WebSocket
```powershell
python -m backend.server.app --port 3210
# 验证：日志应显示 "WebSocket server started on 127.0.0.1:3210"
```

终端 3：前端
```powershell
cd TutorGridFront
yarn dev:web   # 仅 Web，浏览器访问；不会自动 spawn 后端
# 或
yarn dev       # Electron，Electron 会再 spawn 一份后端（端口冲突就用 dev:web）
```

## 五、F04 验收测试清单

### 测试 1：协议 smoke ✅ 必测

**目的**：确认 WebSocket 三个 method（create/resume/interrupt）都能响应。

**步骤**：

1. 开启 Orchestrator 后端（4.2 或 4.3）
2. 打开浏览器 DevTools Console，粘贴：

```js
const ws = new WebSocket("ws://127.0.0.1:3210/ws/orchestrator");
ws.onopen = () => console.log("✅ WS connected");
ws.onmessage = (e) => console.log("📩", JSON.parse(e.data));
ws.onerror = (e) => console.error("❌", e);
```

3. 连接成功后发送 create：

```js
ws.send(JSON.stringify({
  type: "request",
  id: "req_test_001",
  method: "orchestrator.task.create",
  params: {
    session_id: "sess_test",
    doc_id: "hyper_test",
    instruction: "帮我用 sklearn 跑一个线性回归"
  }
}));
```

**预期**：
- ✅ 收到 `{type:"response", id:"req_test_001", ok:true, data:{task_id:"...", status:"pending"}}`
- ✅ 紧接着收到一串 `{type:"event", event:"orchestrator.task.step", ...}` 步骤推送
- ✅ 最后收到 `{type:"event", event:"orchestrator.task.result", ...}`

**失败定位**：
| 现象 | 可能原因 |
|---|---|
| WS 连不上 | 端口 3210 没起，看终端 2 是否有 banner |
| `ok: false` | 缺 `instruction` / `session_id` 字段；看返回的 `error` 字段 |
| 收到 `step` 但卡在 `running` 不动 | LLM 配置问题，看 `backend/config.py` 是否注入了 DeepSeek key |
| 收到 `failed` | LangSmith 抓详情或看 Orchestrator 终端 stack trace |

---

### 测试 2：步骤级流式推送 ✅ 必测

**目的**：确认 `task.step` 按 4 步顺序推送（planning → tools → verify → finalize）。

**步骤**：在测试 1 的 ws.onmessage 里筛选 step 事件：

```js
ws.onmessage = (e) => {
  const m = JSON.parse(e.data);
  if (m.event === "orchestrator.task.step") {
    console.log(`Step ${m.data.step_index}/${m.data.step_total}: ${m.data.phase} = ${m.data.status}`);
  }
};
```

**预期**输出形如：
```
Step 1/4: planning = running
Step 1/4: planning = done
Step 2/4: tools = running
Step 2/4: tools = done
Step 3/4: verify = running
Step 3/4: verify = done
Step 4/4: finalize = running
Step 4/4: finalize = done
```

**通过条件**：
- ✅ phase 字段值在枚举内：`planning | tools | verify | finalize`
- ✅ status 字段值在枚举内：`pending | running | done | failed | awaiting_user | interrupted`
- ✅ step_index/step_total 数字正确（不是 undefined）
- ✅ 顺序符合 planning → tools → verify → finalize（不会跳跃）

---

### 测试 3：中断恢复 awaiting_user ⚠️ 关键

**目的**：确认 LangGraph 原生 `interrupt()` + `Command(resume=...)` 链路通。

**步骤**：发起一个会触发 `await_user` 工具的任务（任务文本里要含"询问用户/选择"等）：

```js
ws.send(JSON.stringify({
  type: "request",
  id: "req_test_002",
  method: "orchestrator.task.create",
  params: {
    session_id: "sess_test",
    doc_id: "hyper_test",
    instruction: "请先问用户使用本地数据还是云端数据，再开始分析"
  }
}));
```

**预期**：
- ✅ 收到一个 `event: "orchestrator.task.awaiting_user"` 事件，含 `prompt` 字段（"请选择数据源..."）
- ✅ 任务**不终止**，处于暂停状态

然后发送 resume：

```js
// 把上面拿到的 task_id 替换进来
ws.send(JSON.stringify({
  type: "request",
  id: "req_test_003",
  method: "orchestrator.task.resume",
  params: {
    task_id: "<上面拿到的 task_id>",
    session_id: "sess_test",
    input: { kind: "reply", content: "用本地数据" }
  }
}));
```

**通过条件**：
- ✅ resume 后任务继续推进，最后给出 `task.result`
- ✅ result 内容能体现用户的回答（"用本地数据"被吸收进了执行）

**失败定位**：
- 如果 resume 没生效（任务没继续走）→ 后端 A 自己说"双路径还没收口"，告诉他 task_id 和时间点

### 测试 4：interrupt 主动取消

**步骤**：开始一个任务，立刻取消：

```js
ws.send(JSON.stringify({
  type: "request",
  id: "req_test_004",
  method: "orchestrator.task.interrupt",

    task_id: "<前面拿到的 task_id>",

  }

```

**通过条件**：
- ✅ 收到 step 事件 status 变为 `interrupted`
- ✅ 不会继续推送后续 step

## 六、F05 验收测试清单

### 测试 6：python_runner 端到端 ⚠️ 答辩保底链路

**目的**：F05 验收点 1 — "帮我跑 sklearn 线性回归 → 返回 stdout + 图片"。

**当前状态**：链路具备，**没有稳定 demo 样例**（后端 A 自己承认）。所以这条测试**会失败或不稳**，但跑一次能让我们知道差什么。

**步骤**：发送一个明确要求生成图片的任务：

```js
ws.send(JSON.stringify({
  type: "request",
  id: "req_test_006",
  method: "orchestrator.task.create",
  params: {
    session_id: "sess_test",
    doc_id: "hyper_test",
    instruction: "用 sklearn 生成 100 个二维数据点，跑线性回归，把拟合图保存为 plot.png"
  }
}));
```

**预期** `task.result` 事件 data 形如：

```json
{
  "task_id": "...",
  "status": "done",
  "result_type": "code_output",
  "content": "✅ R² = 0.94 ...",
  "artifacts": [
    {"type": "image", "path": "data/artifacts/plot.png"},
    {"type": "code", "language": "python", "content": "..."}
  ]
}
```

**通过条件**：
- ✅ `status: "done"`
- ✅ `content` 含真实的 stdout（R² 值或类似）
- ✅ `artifacts` 数组里**至少有 1 张 image** 且 path 文件真实存在

**失败定位**：
- 没有 image artifact → 跟后端 A 反馈："F05 验收点 1 缺图片产物"
- crash → 看终端 stack trace，定位是 Python Runner 还是 LLM 输出格式问题

### 测试 7：Worker 失败 fallback ✅ 必测

**目的**：F05 验收点 2 — "Worker 不可用时有 fallback 提示而非 crash"。

**步骤**：

A. **先确认 codex / opencode CLI 没装**（或临时改 PATH 让它们找不到）：

```powershell
codex --version    # 如果报"找不到命令"就是没装，正合适
opencode --version # 同上
```

B. 发起一个会触发 codex/opencode 的任务（含 `review` / `implement` 关键词）：

```js
ws.send(JSON.stringify({
  type: "request",
  id: "req_test_007",
  method: "orchestrator.task.create",
  params: {
    session_id: "sess_test",
    doc_id: "hyper_test",
    instruction: "review 这段代码：def add(a,b): return a+b"
  }
}));
```

**通过条件**：
- ✅ **不 crash**（后端进程还活着，能继续接受新请求）
- ✅ 收到 `task.result`，`status` 不是 `failed` 而是给了 fallback 内容
- ✅ result 元数据里能看到 `worker: "fallback"` 或 `metadata.fallback_recommended: true`
- ✅ summary 里给出"回退到纯 LLM 推理"提示

## 七、故障排查决策树

```
任务发起后无任何响应
├── WS 连不上？
│   └── 端口 3210 没起 → 启动 backend.server.app
├── 收到 response 但 ok: false？
│   └── 看 error 字段；通常是参数缺失
├── 收到 step 但卡在 planning?
│   └── LLM 配置问题：检查 backend/config.py + .env DEEPSEEK_API_KEY
├── 卡在 tools?
│   └── Worker 调用问题：检查 codex/opencode/python 是否在 PATH
├── 收到 failed status?
│   └── 看后端终端 stack trace；常见：
│       - sklearn 没装 → pip install scikit-learn
│       - matplotlib 没装 → pip install matplotlib
│       - 工作区路径错 → 检查 DEFAULT_WORKSPACE_ROOT
└── 收到 done 但前端不刷新?
    └── 前端 store 订阅问题，看 orchestratorTaskStore.ts
```

## 八、已知缺口（按后端 A 自己说的）

来自 [harness/docs/current-focus.md](../../harness/docs/current-focus.md) 和 [worker-delegation.md](../worker-delegation.md)：

| 缺口 | 影响 | 谁来补 |

| `interrupt/resume` 双路径还没压成单一路径 | 中断恢复偶尔可能有兼容问题 | 后端 A |
| 原生 `interrupt/resume` 缺集成测试 | 回归风险 | 后端 A |
| LangGraph stream 结构化事件还可以更细 | 前端进度条粒度不够细 | 后端 A |
| **F05 验收点 1 没有稳定 sklearn demo 样例** | **答辩演示风险点** | 后端 A + 前端 A 配合 |
| `task.result` 字段稳定性还没强制约定 | 前端解析可能脆 | 后端 A |
